零件箱
MCP、Skills、子代理

你在網路上聽到的那堆縮寫，其實都是為了解決 M1 那個箱子的問題。一個一個拆給你看。

1MCP：往箱子裡插工具的標準插頭

全名 Model Context Protocol。名字嚇人，概念其實就是 USB。

沒有 MCP 之前的世界：你有 3 個 AI 工具，想接 4 個資料來源（Google Drive、Slack、資料庫、GitHub），你得寫 3 × 4 = 12 組客製串接。每多一個工具或來源，數量就爆炸。

有了 MCP 之後：大家講好一套共同的接法。每個資料來源只要實作一次 MCP，任何支援 MCP 的 AI 都能接。3 + 4 = 7 組就夠了。

這就是 USB 幹過的事：在 USB 之前，滑鼠、鍵盤、印表機各有各的接頭；USB 出現後，一個孔全解決。MCP 是 Anthropic 在 2024 年 11 月提出的，現在已經變成跨廠商的通用標準。

2Skills：需要時才載入的說明書

直接回到 M1 的箱子問題：說明書很佔空間，但你多數時候用不到。

假設你想教 AI 一百件事：怎麼寫你們公司的報告格式、怎麼操作實驗室儀器、怎麼產出投影片……

笨方法：把一百份說明書全部塞進箱子。結果箱子還沒開始工作就滿了，而且它 99% 的時間都用不到那些。

Skills 的做法（漸進式揭露）：箱子裡平常只放一百個標題（「這份是講報告格式的」），大概幾十個字。等模型判斷「這次真的需要報告格式」，harness 才把那一份完整內容載進箱子。

這跟你查書一模一樣：你不會把整本教科書背下來，你只記得目錄，需要時才翻到那一頁。 Skill 本身就是一個資料夾，裡面放一份說明文件，可能再附幾個腳本——它不是模型的一部分，是 harness 的一部分。

3Sub-agent：開第二個箱子

這是整門課最實用的一招，也是「為什麼子代理能省 token」的答案。

想像你要 AI 做一件事：「研究我們這個系統怎麼運作，然後改一行程式。」

「研究」這個動作會讀掉一大堆檔案。問題是——那些檔案內容，在真正動手改的時候幾乎都用不到了，你只需要結論。但它們已經佔滿了箱子，而且會一路跟著你到對話結束，因為每輪都要重寄（還記得 M1 嗎）。

子代理的做法：派一個分身去做研究。分身有自己獨立的箱子，它在那裡讀個過癮，讀完只把一小段結論交回主箱子。那一大堆檔案內容，從頭到尾沒有進過主箱子。

互動 · 子代理到底省在哪

任務要做三輪研究，每輪讀掉約 6,000 tokens 的檔案。按「跑一次」，看兩邊的主箱子差多少。

✗ 不用子代理
已用 3,000主箱子 40,000

全部檔案內容都塞在主箱子裡。

✓ 用子代理
已用 3,000主箱子 40,000

主箱子只收到結論。

按上面的按鈕開始。

注意，這裡沒有魔法。子代理讀那 18,000 tokens 的錢還是要付——它在自己的箱子裡照樣消耗。 省下來的不是「總花費」，而是主箱子的空間。而因為主箱子每一輪都要重寄， 留在裡面的東西會被重複計費很多次；讓它保持乾淨，長遠下來才是真的省，而且主對話能撐更久不失憶。

4Multi-agent：很多箱子，以及它的代價

把子代理的想法推到極致：讓好幾個 agent 分工，一個查資料、一個寫程式、一個審查，彼此交談。這就是 multi-agent（多代理）。

聽起來很威，但要先破除一個迷思：

常見說法實務上
多代理「是 agent 演化的完全體，一定更強」 是一種取捨，不是升級。代理之間要溝通、要傳遞脈絡，常常更脆弱、更難除錯、更貴。
什麼時候值得「任務很複雜就該用」 當子任務彼此獨立、可以平行時最有價值（例如同時查五個不相干的主題）。若步驟環環相扣，單一 agent 把迴圈跑穩通常更好。

一個實用的判斷法：如果你沒辦法清楚說出「每個代理各自負責什麼、彼此要交換什麼」，那你需要的多半不是多代理，而是把單一 agent 的迴圈設計好。

名詞小整理： Sub-agent（子代理）是主 agent 派出去的分身，做完就回報、然後消失，是從屬關係； Multi-agent（多代理）泛指多個 agent 協作的架構，可能是平行的同儕。 兩者常被混用，但骨子裡都是同一招：用額外的箱子，換主箱子的乾淨。
