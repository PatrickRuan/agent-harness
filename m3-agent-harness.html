Module 3 · 核心（本課最長，慢慢看）

Agent = Model + Harness

模型是被關在密室裡的大腦。這一課從「30 行 Python」開始，一路講到它怎麼長出手腳、學會繞圈。

Model＋ Harness＝ Agent

1先把兩邊分清楚

這條界線，是理解整個 AI 應用世界的鑰匙。

Model ＝ 大腦

一顆訓練好的神經網路。收到文字，吐出文字。

  • 理解意圖、推理
  • 決定「下一步該做什麼」
  • 本身不會上網、不存記憶、不執行任何動作

Harness ＝ 骨架＋神經＋工具箱

圍繞模型的那層程式碼。你寫的，或框架幫你寫的。

  • 組裝要餵進箱子的脈絡（M1 那個箱子）
  • 派發並實際執行工具
  • 跑迴圈、重試、控制權限與成本

模型像一顆很聰明、但被關在密室裡的大腦——能思考，但沒有手、沒有眼睛、也記不住昨天。Harness 就是接上去的那副身體：眼睛（讀資料）、手腳（呼叫工具）、神經（把訊息傳來傳去），還有那條「不准做危險事」的紅線。

接下來我們照著歷史順序，看這副「身體」怎麼從 30 行程式，長成今天能自己工作好幾小時的樣子。

2最原始的樣子：20–30 行 Python 聊天

早期我們剛拿到 LLM API，寫的程式其實就是「第一代 harness」——最輕薄的一層。

那時候模型只是個「文字接龍機」：你給它一串對話，它接下一句。那 30 行程式碼只負責三件小事：

1. 設定 system prompt（角色） 2. 把對話堆進 messages 陣列 3. 呼叫 API、印出回覆

關鍵觀念（就是 M1 講的那件事）：模型沒有記憶。它會「記得」前面聊過什麼，唯一的原因是——你每一輪都把整串 messages 再寄一次給它。維護這串陣列，就是 harness 最原始的工作。

互動 · 看 messages 陣列怎麼長大

打一句話送出，看看 harness 幫你維護的對話陣列多了哪兩則。（純前端模擬，不會真的連網）

chat.py — 最原始的 harness（真實可跑）
# pip install openai
from openai import OpenAI

client = OpenAI(api_key="sk-...")  # ← 換成你自己的金鑰

# harness 工作 1：設定角色，並開一個對話陣列
messages = [
    {"role": "system", "content": "你是一個友善、簡潔的中文助理。"},
]

while True:
    user = input("你：")
    if user.strip() in {"exit", "quit"}:
        break

    # harness 工作 2：把使用者這句話堆進陣列
    messages.append({"role": "user", "content": user})

    # harness 工作 3：把「整串」對話寄給模型（模型沒有記憶！）
    resp = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=messages,
    )
    reply = resp.choices[0].message.content
    print("AI：", reply)

    # 把 AI 的回覆也存回陣列，下一輪才接得上
    messages.append({"role": "assistant", "content": reply})

沒學過 Python 完全不用擔心——你只要看懂那三行註解在講什麼就夠了。 補充：更早期（2023 年 GPT-3.5 剛開放）的寫法是 openai.ChatCompletion.create(...)， 概念一模一樣，只是新版 SDK 改成先建一個 client。無論哪種，這整段就是最小的 harness。

3裝上手腳：Tool use ＝ 這才叫 Agent

當我們不滿足於純聊天，想讓它查資料、算數、發信——就得幫模型裝上「手腳」。

模型自己不會上網查天氣。它能查，是因為 harness 做了一套「傳話」：

  1. Harness 用 JSON 把「有哪些工具、怎麼用」描述給模型看。
  2. 模型看懂後說：「我要呼叫 get_weather，參數是台北」——它只是輸出一段文字，並沒有真的執行。
  3. Harness 攔截這段文字，實際去跑程式、連網、拿到氣溫。
  4. Harness 把結果放回箱子，再呼叫一次模型，模型才組出人話回答你。
最容易搞混的一點：模型負責「決定要做什麼」，harness 負責「真的去做」。工具的程式碼是你寫在 harness 裡的，不在模型裡面。模型只是說出它想呼叫誰。
互動 · 一步步看 Agent 回答「台北現在幾度？」

按「下一步」，把模型與 harness 之間的來回攤開來看。

🧠 Model 思考
使用者問台北氣溫，我沒有即時資料，需要呼叫工具。
🧠 Model 輸出工具呼叫
送出 get_weather({"city":"台北"})（注意：這只是文字，還沒真的執行）
⚙️ Harness 執行
攔到呼叫 → 實際跑程式打氣象 API…
⚙️ Harness 把結果放回箱子
{"temp":32,"desc":"晴"} 塞回對話，再呼叫一次模型。
🧠 Model 最終回答
「台北現在大約 32°C，天氣晴，記得補水 ☀️」
agent.py — 幫模型裝上一個工具（真實可跑，節錄）
# harness 工作：用 JSON 描述「有哪些工具」給模型看
tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "查詢某城市目前氣溫",
        "parameters": {"type":"object",
            "properties":{"city":{"type":"string"}},
            "required":["city"]},
    },
}]

def get_weather(city):        # ← 工具的「真身」寫在 harness 裡
    return {"temp": 32, "desc": "晴"}   # 真實情況會去打氣象 API

messages = [{"role":"user","content":"台北現在幾度？"}]
resp = client.chat.completions.create(
    model="gpt-4o-mini", messages=messages, tools=tools)

msg = resp.choices[0].message
if msg.tool_calls:                     # 模型「決定」要用工具
    call = msg.tool_calls[0]
    args = json.loads(call.function.arguments)
    result = get_weather(**args)        # harness「真的」執行

    # 把結果餵回去，再問一次模型 → 這就是「迴圈」的一圈
    messages.append(msg)
    messages.append({"role":"tool",
        "tool_call_id":call.id,
        "content":json.dumps(result)})
    final = client.chat.completions.create(
        model="gpt-4o-mini", messages=messages)
    print(final.choices[0].message.content)

4靈魂不是工具，是迴圈

很多人以為「有工具就是 Agent」。更準確地說：Agent 是會自己一圈一圈跑到完成的東西。

給電機系的類比：這就是回授控制系統。
你們之後會學到的 sense → decide → actuate → 再 sense，跟 agent 的 想 → 動手 → 看結果 → 再想 是同一個結構。 模型是控制器，工具是致動器，工具回傳的結果是感測器讀值，任務完成條件就是設定點（setpoint）。 沒有回授的 agent，就是開迴路控制——你只能祈禱它一次做對。

純聊天是「你問一句、它答一句」。Agent 會反覆：查完發現要改條件、算錯了自己重算、寫完程式跑測試沒過就回頭修——直到達標或撞到你設的上限。

沒完成就回到「想」，再繞一圈。這個迴圈就是 harness 的心臟。

真實情況：它常常會失敗、然後自己修

教科書的迴圈圖看起來很漂亮，實際跑起來大多長這樣：

第 1 圈 · 想
我要修這個 bug，先改這個檔案。
第 1 圈 · 動手
改完，跑測試。
第 1 圈 · 看結果 ✗
測試沒過，跳出三個錯誤。
第 2 圈 · 想
喔，我漏掉一個相依。改別的地方。
第 2 圈 · 看結果 ✓
測試全過，收工。

這種「自己撞牆、自己爬起來」的能力，正是 agent 和聊天機器人最大的差別。而什麼時候該放棄別再撞了，就是 M2 講的 loop engineering 在管的事。

5為什麼 Harness 最近變「顯學」？

模型單靠自己撞牆後，大家把心力放到「外面那層程式」，harness 才被推上檯面。

過去 harness 只是個「跑腿的」——傳傳資料而已。現在大家發現：直接叫模型硬答，效果常常不夠好；但在外面那層加上一些技巧，同一個模型就能穩很多：

脈絡管理（塞對的東西進箱子） 失敗重試 / 自我修正 反思 reflection RAG 檢索增強 子 agent 分工
互動 · 同一個模型，兩種 harness

任務：「幫我查我們 Q2 的退貨率，並說明為什麼變高」。切換看看兩種 harness 餵給同一個模型時，出來的東西差多少。

別誤會：「好 harness 能讓中階模型贏過高階模型」這句話講得太滿。 更準的說法是——同一個模型在不同 harness 下，表現天差地遠；harness 的品質，有時和模型本身一樣重要。 但硬要用陽春 harness 配弱模型去贏強模型，通常不成立。

那 LangChain、AutoGen、CrewAI 是什麼？

它們是 framework（框架）——幫你「蓋 harness」的工具箱，不等於 harness 本身。你也可以完全不用框架，自己用 30 行寫一個 harness（很多正式產品就是這樣，為了可控）。

6把話講精準（幾個常見的小誤會）

這些說法「聽起來對、但不夠精準」，這裡校準一下。

主題常見／含糊的說法更精準的說法
名詞的年紀「早期那 30 行就叫 harness」 那層薄 wrapper 的概念早就有；但 harness 這個詞被廣泛使用，是 2024–2026 才發生的事。
Agent 的定義「有工具＝Agent」 靈魂是迴圈：想→做→看→再想，反覆到完成。工具只是迴圈裡的一步。
Harness 的威力「中階模型靠 harness 就能贏高階模型」 太滿。準確版：同一模型在不同 harness 下差很多；harness 品質有時和模型一樣重要。
框架 vs harness「LangChain＝harness」 它們是 framework（幫你蓋 harness 的工具），不等於 harness 本身。
MCP 是什麼「MCP 是多 agent 協作架構」 MCP 是一個連接工具／資料給模型的協定，像 USB 標準口，不是協作架構。（詳見 M4）
Multi-agent「多 agent 是 harness 的完全體」 它是一種架構取捨，常更脆弱、更難除錯，不代表更高級。單 agent 把迴圈做穩，往往更實用。
最吃重的活（常被忽略） 現代 harness 最花心思的其實是脈絡管理：箱子有限，每輪要決定塞什麼、丟什麼、壓縮什麼。
本課總結：Agent = Model + Harness 這個公式在今天的語境下站得住腳，也很有代表性。 但真正的靈魂是那個迴圈——沒有迴圈的工具呼叫，只是比較花俏的問答。
