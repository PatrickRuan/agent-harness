白話拆解

從「聊天玩具」到 Agent
Harness 到底是什麼？

一頁把「模型」跟「外面那層程式」的界線講清楚。可以親手撥動的互動教學。

Model ＋ Harness ＝ Agent

1一句話：Agent = Model + Harness

先給結論，再慢慢拆。這條界線是理解整個 AI 應用世界的鑰匙。

Model ＝ 大腦

一顆訓練好的神經網路。收到文字，吐出文字。

  • 理解意圖、推理
  • 決定「下一步該做什麼」
  • 本身不會上網、不會存記憶、不會執行任何動作

Harness ＝ 骨架＋神經＋工具箱

圍繞著模型的那層程式碼。你寫的、或框架幫你寫的。

  • 組裝要餵給模型的脈絡（context）
  • 派發並實際執行工具（tools）
  • 跑迴圈（loop）、重試、控制邊界與安全

一個比喻

模型像一顆很聰明、但被關在密室裡的大腦——它能思考，但沒有手、沒有眼睛、也記不住昨天。Harness 就是接到這顆大腦上的身體：眼睛（讀資料）、手腳（呼叫工具）、脊椎與神經（把訊息傳來傳去、跑迴圈），還有那條「不准做危險事」的紅線。

這一頁接下來，就是照著歷史順序，看這副「身體」怎麼從 30 行長成今天的樣子。

2最原始的樣子：20–30 行 Python 聊天

早期我們剛拿到 LLM API，寫的程式其實就是「第一代 harness」——最輕薄的一層。

那時候模型只是個「文字接龍機」：你給它一串對話，它接下一句。那 30 行程式碼只負責三件小事：

1. 設定 system prompt（角色） 2. 把對話堆進 messages 陣列 3. 呼叫 API、印出回覆

關鍵觀念：模型沒有記憶。它會「記得」前面聊過什麼，唯一的原因是——你每一輪都把整串 messages 再寄一次給它。維護這串陣列，就是 harness 最原始的工作。

互動 · 看 messages 陣列怎麼長大

打一句話送出，看看 harness 幫你維護的對話陣列多了哪兩則。（純前端模擬，不會真的連網）

chat.py — 最原始的 harness（真實可跑）
# pip install openai
from openai import OpenAI

client = OpenAI(api_key="sk-...")  # ← 換成你自己的金鑰

# harness 工作 1：設定角色，並開一個對話陣列
messages = [
    {"role": "system", "content": "你是一個友善、簡潔的中文助理。"},
]

while True:
    user = input("你：")
    if user.strip() in {"exit", "quit"}:
        break

    # harness 工作 2：把使用者這句話堆進陣列
    messages.append({"role": "user", "content": user})

    # harness 工作 3：把「整串」對話寄給模型（模型沒有記憶！）
    resp = client.chat.completions.create(
        model="gpt-4o-mini",
        messages=messages,
    )
    reply = resp.choices[0].message.content
    print("AI：", reply)

    # 把 AI 的回覆也存回陣列，下一輪才接得上
    messages.append({"role": "assistant", "content": reply})

補充：更早期（2023 GPT-3.5 剛開放）寫法是 openai.ChatCompletion.create(...)，概念一模一樣，只是新版 SDK 改成先建 client。無論哪種，這整段就是最小的 harness。

3裝上手腳：Tool use ＝ 這才叫 Agent

當我們不滿足於純聊天，想讓它查資料、算數、發信——就得幫模型裝上「手腳」。

模型自己不會上網查天氣。它能查，是因為 harness 做了這套「傳話」：

  1. Harness 用 JSON 把「有哪些工具、怎麼用」描述給模型看。
  2. 模型看懂後說：「我要呼叫 get_weather，參數是台北」——它只是輸出一段 JSON，並沒有真的執行。
  3. Harness 攔截這段 JSON，實際去跑 Python、連網、拿到 25°C。
  4. Harness 把結果餵回給模型，模型才組出人話回答你。
重點：模型負責「決定要做什麼」，harness 負責「真的去做」。工具的程式碼是你寫在 harness 裡的，不在模型裡。
互動 · 一步步看 Agent 怎麼回答「台北現在幾度？」

按「下一步」，把模型與 harness 之間的來回攤開來看。

🧠 Model 思考
使用者問台北氣溫，我沒有即時資料，需要呼叫工具。
🧠 Model 輸出 tool_call
送出 JSON：get_weather({"city":"台北"})（注意：只是文字，還沒真的執行）
⚙️ Harness 執行
攔到 tool_call → 實際跑 Python 打氣象 API…
⚙️ Harness 餵回結果
{"temp":32,"desc":"晴"} 塞回對話陣列，再呼叫一次模型。
🧠 Model 最終回答
「台北現在大約 32°C，天氣晴，記得補水 ☀️」
agent.py — 幫模型裝上一個工具（真實可跑，節錄）
# harness 工作：用 JSON schema 描述「有哪些工具」
tools = [{
    "type": "function",
    "function": {
        "name": "get_weather",
        "description": "查詢某城市目前氣溫",
        "parameters": {"type":"object",
            "properties":{"city":{"type":"string"}},
            "required":["city"]},
    },
}]

def get_weather(city):        # ← 工具的「真身」寫在 harness 裡
    return {"temp": 32, "desc": "晴"}   # 真實情況會去打氣象 API

messages = [{"role":"user","content":"台北現在幾度？"}]
resp = client.chat.completions.create(
    model="gpt-4o-mini", messages=messages, tools=tools)

msg = resp.choices[0].message
if msg.tool_calls:                     # 模型「決定」要用工具
    call = msg.tool_calls[0]
    args = json.loads(call.function.arguments)
    result = get_weather(**args)        # harness「真的」執行

    # 把結果餵回去，再問一次模型 → 這就是「迴圈」的一圈
    messages.append(msg)
    messages.append({"role":"tool",
        "tool_call_id":call.id,
        "content":json.dumps(result)})
    final = client.chat.completions.create(
        model="gpt-4o-mini", messages=messages)
    print(final.choices[0].message.content)

4Agent 的靈魂：迴圈（loop）

很多人以為「有工具就是 Agent」。更準確地說：Agent 是會自己一圈一圈跑到完成的東西。

純聊天是「你問一句、它答一句」。Agent 不一樣——它會：想 → 動手 → 看結果 → 再想 → 再動手…直到任務完成。上一章的「工具」只是繞一圈；真正的 agent 會繞很多圈：查完天氣發現要換城市、算錯了自己重算、寫完程式跑測試沒過就修。

🧠 想think
→
✋ 動手act
→
👀 看結果observe
↩
✅ 完成？done?

沒完成就回到「想」，再繞一圈。這個 while 迴圈，就是 harness 的心臟。

你正在用的 Claude Code、Cursor、各種「AI 幫我把事情做完」的工具，底層都是這個迴圈：模型提議一個動作 → harness 執行（讀檔、跑指令、改程式）→ 把結果貼回去 → 模型看了再決定下一步。

5為什麼 Harness 最近變「顯學」？

你的直覺沒錯：模型單靠自己撞牆後，大家把心力放到「外面那層程式」，harness 才被推上檯面。

過去 harness 只是個「跑腿的」——傳傳資料而已。現在大家發現：直接叫模型硬答，效果常常不夠好；但在外面那層加上一些技巧，同一個模型就能穩很多：

脈絡管理（塞對的東西進 context） 失敗重試 / 自我修正 反思 reflection RAG 檢索增強 子 agent 分工
互動 · 同一個模型，兩種 harness

任務：「幫我查我們 Q2 的退貨率，並說明為什麼變高」。切換看看兩種 harness 餵給同一個模型時，出來的東西差多少。

別誤會：「好 harness 能讓中階模型贏過高階模型」這句話講得太滿。更準的說法是——同一個模型在不同 harness 下，表現天差地遠；harness 的品質，有時和模型本身一樣重要。但硬要用陽春 harness 配弱模型去贏強模型，通常不成立。

那 LangChain、AutoGen、CrewAI 是什麼？

它們是 framework（框架）——幫你「蓋 harness」的工具箱，不等於 harness 本身。你也可以完全不用框架，自己用 30 行寫一個 harness（很多正式產品就是這樣，為了可控）。

6把話講精準（幾個常見的小誤會）

你的整體脈絡是對的。這裡把幾句「聽起來對、但不夠精準」的說法校準一下。

主題常見/含糊的說法更精準的說法
名詞的年紀 「早期那 30 行就叫 harness」 那層薄 wrapper 的概念早就有；但 harness 這個詞被廣泛使用，是近一兩年（2024–2025）在 agent／eval／coding agent 圈才流行。你「以前就有、最近才顯學」的直覺很準。
Agent 的定義 「有工具＝Agent」 靈魂是迴圈：想→做→看→再想，反覆到完成。工具只是迴圈裡的一步。
Harness 的威力 「中階模型靠 harness 就能贏高階模型」 太滿。準確版：同一模型在不同 harness 下差很多；harness 品質有時和模型一樣重要。
框架 vs harness 「LangChain＝harness」 它們是 framework（幫你蓋 harness 的工具），不等於 harness 本身。
MCP 是什麼 「MCP 是多 agent 協作架構」 MCP 是一個連接工具/資料給模型的協定（protocol），像 USB 標準口，不是協作架構。
Multi-agent 「多 agent 是 harness 的完全體」 它是一種架構取捨，常更脆弱、更難除錯，不代表更高級。單 agent 把迴圈做穩，往往更實用。
最吃重的活 （常被忽略） 現代 harness 最花心思的其實是脈絡管理：context window 有限，每一輪要決定塞什麼、丟什麼、壓縮什麼。
結論：你的四段推論方向都對，Agent = Model + Harness 這個公式在今天的語境下站得住腳、也很有代表性。上面只是把幾個轉角磨得更精準而已。

7這一頁是怎麼做出來的

呼應參考頁的精神：把製作過程也攤開。這頁本身，就是一個「人給目標、AI 跑迴圈」的小例子。

1 · 用 Claude Code 的 Plan Mode 先規劃

沒有一開始就寫 code。先讓 AI 讀參考頁 mnist-knob-lab 抓風格、確認「互動程度」與「程式範例怎麼處理」兩個關鍵選擇，寫成一份計畫、經人核准後才動工。

2 · 單一自包含 HTML 的取捨

整頁就一個 index.html，HTML／CSS／JS 全部 inline，零外部相依。好處：可離線打開、可直接丟上 GitHub Pages、不怕某個 CDN 掛掉。代價：檔案較長，但對教學頁很划算。

3 · 為什麼用「前端模擬」而不是真的連 OpenAI

頁面上的三個互動（messages 堆疊、Agent 步進器、harness 對比）都是純前端演的。因為：不該把 API 金鑰放進公開網頁，而且教學重點是「看懂流程」，模擬就夠。真正可跑的程式碼另外附在 ② ③ 章，讓你複製回自己電腦、填上自己的金鑰去跑。

4 · 內容的事實把關

製作時不是照單全收原始說法，而是把「常見但不精準」的幾句話挑出來校準（就是第 ⑥ 章那張表）。AI 生成很方便，但脈絡與名詞的精準度需要人＋模型一起把關。
